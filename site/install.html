Setup guide

Install once, then run repeatable genome workflows.

WGS Extract CLI uses Pixi to manage Python, the app itself, and many external bioinformatics tools. For macOS and Linux, the recommended installer wraps Pixi into a standalone directory with CLI and GUI launchers.

QuickstartDeveloper Pixi setupPlatformsReferencesConfigurationVerify

Run the installer

On macOS or Linux, pipe the installer into sh or download install.sh and double-click/run it from your Downloads folder.

Use the standalone folder

The app, Pixi environment, cache, CLI launcher, and GUI launchers live under one wgsextract-cli/ directory.

Bootstrap references

Download the metadata and reference-library structure WGS Extract uses to find FASTA, VCF, ploidy, liftover, and annotation assets.

Run a small test

Use info and region-limited commands first. Whole-genome jobs can take hours and hundreds of gigabytes of working space.

macos-linux.shrecommended installer
# Install into ./wgsextract-cli
curl -fsSL https://raw.githubusercontent.com/theontho/wgsextract-cli/main/install.sh | sh

# Initialize reference library data
./wgsextract-cli/bin/wgsextract ref bootstrap
./wgsextract-cli/bin/wgsextract ref library --list
./wgsextract-cli/bin/wgsextract ref library --install hs38

# Verify the app and environment
./wgsextract-cli/bin/wgsextract info --detailed
./wgsextract-cli/bin/wgsextract deps check

# Launch the desktop GUI
./wgsextract-cli/start-wgsextract-gui.sh

Developer Pixi setup

Technical users who want a normal source checkout, editable code, Git branches, or direct Pixi environment control should use Pixi directly.

Best for contributors

Use this path when you plan to edit code, run tests, inspect package metadata, or switch branches frequently.

Same toolchain

The standalone installer still uses Pixi underneath. The developer setup just leaves Pixi and the source checkout exposed.

developer-pixi.shtechnical users
# Install Pixi first
curl -fsSL https://pixi.sh/install.sh | bash

# Clone and install from source
git clone https://github.com/theontho/wgsextract-cli.git
cd wgsextract-cli
pixi install

# Run through Pixi
pixi run wgsextract --help
pixi run wgsextract deps check
pixi run wgsextract gui --desktop

Platform notes

Bioinformatics tools are most mature on Unix-like environments. Windows is supported best through WSL2, with native Windows workflows available for some toolchains.

macOS

Use the standalone installer by default. It creates a wgsextract-cli/ folder with Finder-friendly .command GUI launchers.

osx-arm64osx-64

Linux and WSL2

Use the standalone installer by default on Linux and inside WSL2. Pixi resolves Python and native command-line tools through conda-forge and bioconda.

linux-64WSL2 recommended

Native Windows

Use the Windows PowerShell Pixi installer for Python/app setup. For native bio tools, use the MSYS2 UCRT64 pacman runtime described in the Windows docs.

Reference library setup

Most genome operations need a reference genome and companion files. The reference library lets WGS Extract resolve those assets instead of asking you for every path on every command.

Bootstrap data

ref bootstrap initializes the library scaffolding and common metadata. Then use the library commands to list and install supported builds such as hs38.

./wgsextract-cli/bin/wgsextract ref bootstrap
./wgsextract-cli/bin/wgsextract ref library --list
./wgsextract-cli/bin/wgsextract ref library --install hs38

Index and verify

FASTA references need indexes and dictionaries before alignment and variant callers can use them efficiently.

./wgsextract-cli/bin/wgsextract ref index --ref /path/to/hs38.fa
./wgsextract-cli/bin/wgsextract ref verify --ref /path/to/hs38.fa
Reference builds matter. hg19/GRCh37, hg38/GRCh38, and T2T coordinates differ. Do not mix BAMs, VCFs, annotation files, and gene coordinates from different builds unless you intentionally lift over or re-run the analysis.

Configuration

Use wgsextract config to view the active config path and defaults. You can set input, output, reference, genome library, thread, memory, and external tool paths once instead of repeating them.

Config locations

macOS: ~/.config/wgsextract/config.toml or Application Support. Linux: ~/.config/wgsextract/config.toml. Windows: under AppData.

Genome library

Set genome_library to a folder with one subfolder per person or sample. Then --genome sample-id can resolve inputs and outputs from that folder.

config.tomlexample
input = "/data/genomes/joe/joe.cram"
outdir = "/data/genomes/joe/out"
ref = "/data/reference/hs38.fa"
genome_library = "/data/genome-library"
threads = 8
memory = "16G"
haplogrep_path = "/usr/local/bin/haplogrep"

Verify before running full genomes

Start with environment checks and tiny regions. It is much cheaper to catch missing references, unsorted BAMs, or command syntax errors on chrM than halfway through a full-genome job.

verification-recipes.shsafe first commands
./wgsextract-cli/bin/wgsextract deps check
./wgsextract-cli/bin/wgsextract --input sample.cram info --detailed
./wgsextract-cli/bin/wgsextract --input sample.bam extract mito-vcf --region chrM
./wgsextract-cli/bin/wgsextract --input sample.bam microarray --formats 23andme_v5 --region chrM
